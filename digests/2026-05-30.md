## Run notes (important — degraded environment)

- **Feed reachability:** of 11 configured feeds, **10 returned HTTP 403** to `web_fetch` (BleepingComputer, The Record, Unit42, Krebs, DFIR, SentinelOne, Talos, Google TAG, Elastic, Reddit). Only the **Microsoft Security Blog** was reachable, and it supplied the two highest-value, artifact-rich items of the day. Triage was therefore largely blind; the two Microsoft npm campaigns are the genuine qualifying set.
- **Toolchain:** `apt` initially failed (two unrelated third-party PPAs 403'd, breaking `apt update`); repaired by disabling them. Final toolchain verified working: `sigma` 3.0.2 (targets splunk + log_scale), `yara`/`yarac` 4.x, `snort` 2.9.20, `suricata` 7.0.3.
- **Validation:** all 14 committed rule files compile/convert clean — 19 exit-0 checks, 0 failures. The only non-zero is `sigma check`'s runtime MITRE **D3FEND** fetch hitting the sandbox 403 egress block (environment limitation, not a rule defect); the portability oracle (`sigma convert` → Splunk **and** CrowdStrike log_scale) passed exit 0 for every Sigma rule, so they are labeled accordingly.
- **Integrity holds (why two items were dropped):**
  - **Copy Fail / CVE-2026-31431** — the seed identifier (CVE-2026-1015) turned out to be an unrelated IBM SSRF; the researcher pivoted to a post-cutoff Linux page-cache LPE, but the Elastic source and every corroborating page returned 403, so the threat could not be confirmed against any primary source. The research summary (including well-encoded auditd-hex draft rules) is retained as a trail, but **no standalone rules were promoted** — shipping blessed detections for an unverifiable vuln would violate the viability gate.
  - **FortiWeb / CVE-2026-22891** — researcher hit a stream-idle timeout with no draft written; advisory unverifiable. Dropped.
